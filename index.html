SRE-подход на практике

Пример инцидента и постмортема

Инцидент: Telegram-бот перестал отвечать на запросы пользователей из-за дедлока при конкурентной записи в файловый кеш во время пиковой нагрузки.

Влияние: Полная недоступность сервиса для ~50 пользователей в течение 25 минут.

Расследование: Анализ логов показал, что несколько потоков одновременно пытались получить блокировку на один и тот же файл, что привело к взаимной блокировке.

Решение:

  • Краткосрочное: Перезапуск сервиса для снятия блокировки.
  • Долгосрочное: Миграция кеша с файловой системы на in-memory хранилище (Redis) для атомарных операций. Добавлен health-check эндпоинт и настроен алерт в Telegram при его недоступности.

Выводы и Action Items: Проблема не в ошибке разработчика, а в архитектуре. Внедрил нагрузочное тестирование для имитации пиковой нагрузки. Добавил в CI-пайплайн статический анализатор для поиска потенциально блокирующих вызовов. Создал Runbook.

Пример Runbook'а: "Бот не отвечает"

Цель: Быстро восстановить работоспособность бота при потере ответа.

Шаги диагностики:

  1. Проверить алерт-канал: Есть ли активные оповещения от системы мониторинга?
  2. Проверить Health Check: Сделать запрос к /health эндпоинту. Ожидаемый ответ: {"status": "ok"}.
  3. Анализ логов: Посмотреть последние логи контейнера: docker logs --tail 100 bot_container. Искать ошибки, связанные с сетью, базой данных или внешними API.

Шаги по устранению:

  • Если проблема в кеше (Redis): Проверить доступность Redis. При необходимости перезапустить контейнер с кешем.
  • Если недоступен внешний API: Включить режим "только чтение из кеша", уведомить пользователей о временных ограничениях.
  • Если непонятная ошибка: Безопасно перезапустить контейнер с ботом: docker restart bot_container.
